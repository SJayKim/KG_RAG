> 한 줄 요약: 환자의 복약 목록 전체를 받아 식약처 DUR **병용금기**(직접 규칙)뿐 아니라,
> 규칙에 없는 **숨은 전이적 상호작용**(두 약이 같은 CYP 효소를 두고 경쟁 → 제3의 약 농도 상승)까지
> 그래프 순회로 찾아내고, 그 **근거 경로(node-edge path)를 감사 가능한 형태로 출력**하는 LangGraph 에이전트.

## Problem Statement

고령·만성질환 환자는 5~10개 약을 동시 복용한다(polypharmacy). 위험한 상호작용 중 상당수는
어떤 단일 문서·규칙에도 적혀 있지 않다. 예: `amiodarone`과 `clarithromycin`은 **둘 다 CYP3A4를 억제**하고,
같은 처방에 `simvastatin`(CYP3A4 기질)이 있으면 simvastatin 농도가 치솟아 횡문근융해 위험이 커진다.
"amiodarone + clarithromycin + simvastatin 위험"이라는 문장은 **세상 어디에도 없다.** 이건 그래프 위 경로로만 존재한다.

플레인 벡터 RAG는 각 약을 언급한 청크를 따로 가져올 뿐, (a) 모든 약쌍을 열거하거나 (b) 공유 효소 노드를
거쳐 "적혀 있지 않은" 상호작용을 추론하지 못한다. 게다가 NLM이 **2024년 1월 RxNav Drug-Drug Interaction
API를 폐기**해서, 이제는 단순 조회 엔드포인트조차 없다 — 그래프 순회가 사실상 필수가 됐다.

## What Makes This Cool (the "whoa")

- **그래프가 장식이 아니라 답 그 자체.** 출력이 "위험합니다"가 아니라
  `amiodarone —(억제)→ CYP3A4 ←(기질)— simvastatin` 이라는 **실제 경로**다. 면접관이 10초 만에 이해한다.
- **벡터가 구조적으로 못 하는 일을 한다.** 조합적 약쌍 열거 + 공유효소 2~3홉 추론 + 적혀 있지 않은 위험 발견.
- **한국 채용 직결.** 식약처 **공식** DUR 데이터를 쓰고, 성분명 정규화로 한국↔글로벌(RxNorm/ATC) 그래프를
  잇는 cross-ontology 엔지니어링을 보여준다. 영어권 포트폴리오가 거의 안 쓰는 데이터라 차별화가 자동으로 된다.
- **"툴 생태계를 안다"는 신호.** 폐기된 RxNav DDI API의 구멍을 그래프로 메우는 선택 자체가 시그널.

## Constraints

- 솔로, 집중 작업 **2~4주**(목표 3주).
- 데이터는 실제 수집 가능 + 포트폴리오/데모에 라이선스 허용되어야 함(이 프로젝트의 하드 룰).
- CITI 인증/유료 게이트가 필요한 소스(MIMIC, KEGG 벌크 등)는 **사용 금지**.
- 환자 데이터를 쓰지 않는다(거버넌스 리스크 회피). 입력은 **약물 목록 + 인구통계 조건**(연령대/임신 여부)뿐.

## Premises (Phase 3 — 적대적 검증 통과)

1. 식약처 DUR OpenAPI는 **성분쌍 금기를 무료·구조화**해 제공한다. ⚠️ 단 데이터셋이 두 개다 —
   `15059486`(DUR **품목**정보, 제품코드 기반)와 `15056780`(DUR **성분**정보, 성분코드 기반).
   스키마가 `Ingredient↔Ingredient`이므로 **성분 레벨 응답을 주는 ID를 Day1에 확정**해 바인딩한다.
2. RxNav DDI API는 **2024-01-02 폐기**되어 단순 조회 대체재가 없다 — *검증됨* → 그래프 순회의 정당성(단, 이건 보조 논거이고 스파인은 아래 3·4).
3. 위험한 polypharmacy 상호작용의 핵심 클래스(공유 CYP 경쟁)는 **2~3홉 경로로만 표현**되며 벡터 RAG가 못 잡는다 — *검증됨.* (이 프로젝트의 진짜 스파인)
4. 이 문제의 수요자(복약 검토하는 약사/의사)는 명확하고, 한국 채용담당자에게 **즉시 legible**하다 — *검증됨.*
5. "GraphRAG가 좋다"는 **주장만으론 2026년 감점**이다. 벡터 RAG 대비 **측정된 A/B 평가표**가 심장이어야 한다.
   결정적으로, **A/B가 공정하려면 그래프와 벡터가 같은 원문(CC0 라벨 텍스트)을 봐야** 한다 — 차이는 "비밀 데이터"가 아니라 **구조화·순회 능력**에서 나와야 한다.

## Approaches Considered (4개 최종 후보 중 선택)

- **A. PolyPharmGraph (선택)** — 한국 DUR + CYP 전이추론. 한국 채용 직결, 무료 데이터, 진짜 2~3홉. *검증 7.5/10.*
- **B. StareDecisis** — 판례 시효성(법률). 퍼블릭도메인, "그래프가 벡터 이기는 가장 직관적 사례". *7.5/10.*
- **C. MechPath** — 신약재창출 경로(바이오). CC0 완제품 그래프, 데이터 리스크 최저. *7/10.*
- **D. LabelVsReality** — 약물감시 모순탐지. 데이터 엔지니어링 가장 가벼움. *7.5/10.*

## Recommended Approach: A. PolyPharmGraph

목표가 "취업 포트폴리오 + 실제 수요"이고 도메인이 의료다. A는 (1) 식약처 공식 데이터로 한국 채용담당자가
즉시 이해하고, (2) 성분명→RxNorm 근사매칭으로 진짜 cross-ontology 엔지니어링을 보여주며,
(3) 폐기된 RxNav 구멍을 메워 툴 생태계 이해를 증명하고, (4) 전이적 CYP 추론이 1홉 조회가 아닌
진짜 다중홉이라 "그래프가 벡터를 이긴다"를 정직하게 증명할 수 있다. 데이터는 전부 무료/CC0급, 3주 스코프.

### 그래프 스키마 (Neo4j property graph)

**노드** (의도적으로 5종으로 제한 — 미사용 레이어는 컷)
- `Drug` (품목, 품목기준코드) — 한국 제품
- `Ingredient` (성분, 성분코드/ATC) — **조인 키**
- `RxConcept` (RxCUI/ATC) — 글로벌 크로스워크 앵커
- `Enzyme` (CYP3A4, CYP2D6, CYP2C9 …)
- `PatientContext` (연령대 / 임신 등 DUR 조건)

> 컷한 것: `AdverseEvent(MedDRA)`·`Condition(ICD/MONDO)`·`TREATS`·`CAUSES`. MedDRA/ICD 매핑은 그 자체로 무겁고,
> 핵심 4개 에이전트 노드와 평가가 쓰지 않는다. v2로 미룬다(YAGNI).

**엣지**
- `(Ingredient)-[:CONTRAINDICATED_WITH {reason, source:"DUR"}]->(Ingredient)` — 병용금기
- `(Ingredient)-[:DUPLICATE_THERAPY {source:"DUR"}]->(Ingredient)` — 효능군중복주의
- `(Ingredient)-[:RISK_FOR {rule, source:"DUR"}]->(PatientContext)` — 임부금기/노인주의/특정연령대금기/용량주의
- `(Drug)-[:HAS_INGREDIENT]->(Ingredient)`
- `(Ingredient)-[:MAPS_TO {method, confidence}]->(RxConcept)` — 성분명→영문/ATC 근사매칭(아래 Resolver 참조)
- `(Ingredient)-[:SUBSTRATE_OF {source:"openFDA-label"}]->(Enzyme)` / `[:INHIBITS]` / `[:INDUCES]`

**전이적 위험 핵심 Cypher** (적혀 있지 않은 위험을 만드는 패턴)
```cypher
// A가 억제하는 효소를 C가 기질로 쓰면 → C 농도 상승 위험 (둘 다 처방에 있을 때)
MATCH (a:Ingredient)-[:INHIBITS]->(e:Enzyme)<-[:SUBSTRATE_OF]-(c:Ingredient)
WHERE a.code IN $regimen AND c.code IN $regimen AND a <> c
RETURN a.name AS inhibitor, e.name AS enzyme, c.name AS affected_substrate
```

### 데이터 수집 계획 (라이선스 포함)

| 레이어 | 소스 | 접근 | 라이선스 | 비고 |
|---|---|---|---|---|
| 직접 금기 엣지 | 식약처 DUR OpenAPI (병용금기·노인주의·임부금기·효능군중복·용량주의) `15056780`(성분) 또는 `15059486`(품목) | 무료 dev key 자동승인, ~10k/일 | KOGL Type1 (출처표시, 상업·파생 허용) | **이미 성분쌍 엣지 = 거의 추출 0.** Day1에 성분레벨 ID 확정 |
| 한국 약물 텍스트 | e약은요 OpenAPI `data.go.kr 15075057` | 무료 key | KOGL Type1 | 효능/주의/상호작용 텍스트 |
| 이름→글로벌 정규화 | RxNorm + RxNav REST `rxnav.nlm.nih.gov` | **키 불필요** | 공개(NLM) | 성분**명**→RxCUI/ATC 근사매칭 (직접 코드맵 아님 — Resolver 참조) |
| **CYP 기전 엣지 (1순위)** | openFDA Drug Label `api.fda.gov/drug/label.json` | 무료(선택 key) | **CC0** | "Clinical Pharmacology / Drug Interactions" 섹션 → **LLM로 substrate/inhibitor/inducer 추출** |

**CYP 기전 엣지 — openFDA 라벨(CC0) 추출이 1순위. 이게 라이선스와 평가공정성을 동시에 푼다:**
- **왜 DrugBank가 아니라 openFDA인가:** DrugBank ToU는 재배포·파생·타 데이터 혼합을 금지한다. 공개 라이브 데모 +
  공개 repo는 정확히 그 셋을 위반하므로 **이 프로젝트의 하드룰을 깬다.** openFDA는 CC0라 공개 배포가 자유롭다.
- **평가공정성 보너스:** CYP 엣지를 라벨 텍스트에서 추출하면, 벡터 베이스라인도 **같은 라벨 텍스트**를 읽는다.
  따라서 벡터가 전이 케이스를 놓치는 건 "비밀 데이터가 없어서"가 아니라 **순회를 못 해서**다 → A/B가 정직해진다.
- **construction-from-text 쇼케이스:** 비정형 라벨 → 타입 엣지 추출은 채용 리서치가 희소·고평가로 꼽은 역량이다.
- **비공개 QA 교차검증용으로만(절대 배포·커밋 금지):** DrugBank `<enzymes>`, Flockhart CYP450 Table은
  **로컬에서 추출 정밀도를 검증**하는 레퍼런스로만 쓴다. 그래프/데모/repo에는 들어가지 않는다.
  PharmGKB(CC BY-SA)는 share-alike copyleft라 혼합 시 전체에 BY-SA가 전염되므로 **그래프에 넣지 않는다**(필요시 비공개 참조만).

> 핵심: 한국 측 DUR이 "무엇이 위험한가"(공식 규칙)를, openFDA 라벨이 "왜 위험한가"(CYP 기전)를 제공하고,
> 둘을 **성분명 정규화**로 잇는다. 이 크로스워크 + 라벨 추출이 엔지니어링 쇼피스다.

### 에이전트 아키텍처 (LangGraph 상태머신)

명시적 planning vs execution 노드 + 조건부 라우팅(= 2026 채용이 보는 "agentic"):

1. **Resolver 노드** — 약명/제품 → 성분코드 → **성분명(한글)** → **영문/ATC 정규화** → RxNav `approximateTerm`/`rxcui?name=`
   **근사매칭** → 그래프 노드. ⚠️ `성분코드→RxCUI` 직접 코드맵은 **존재하지 않는다**(RxNorm에 한국 코드 없음).
   이름 정규화 + 퍼지매칭이 실제 경로이고, **매칭 성공률(coverage%)을 측정·기록**한다(매핑 실패 자체가 정직성 신호).
2. **Direct-DDI 노드** — 처방 집합 내 모든 DUR 병용금기/효능군중복 엣지 열거 (Cypher).
3. **Transitive-DDI 노드** — `(a)-[:INHIBITS]->(e)<-[:SUBSTRATE_OF]-(c)` 2홉 패턴 순회.
4. **PatientContext 노드** — 연령/임신 조건에 걸리는 DUR 규칙만 필터(엣지 속성 필터링).
5. **Router** — free-text 경고가 더 필요하면 openFDA 라벨/벡터 retrieval 도구 호출 결정.
6. **Synthesis 노드** — 각 플래그를 **근거 경로 + 엣지별 출처(DUR / openFDA-label)**와 함께 출력.
   최종 리포트에 **"약사 검토 필요(not a clinical decision)" 정적 배너**를 붙인다(도메인 신뢰 신호; 무거운 interrupt/resume는 안 씀 — YAGNI).

### Evaluation Harness (프로젝트의 심장)

- **골드셋 ~40문항**, 세 버킷:
  - (a) **직접 금기**(1홉) — 두 파이프라인 모두 맞혀야 함(베이스라인 sanity).
  - (b) **전이적 공유효소 위험**(2~3홉) — **그래프가 잡고 벡터가 놓치는** 헤드라인 케이스.
  - (c) **안전 처방**(negatives) — false-positive율 측정.
- **공정성 규칙 1 — 같은 원문:** 베이스라인 벡터 RAG는 **그래프 CYP 엣지를 추출한 바로 그 openFDA 라벨 텍스트**(+ DUR 규칙문 + e약은요)를
  chunk→embed→retrieve→answer 한다. 그래프와 벡터가 **동일 정보**를 보므로, 차이는 정보량이 아니라 **순회·구조화 능력**에서만 나온다.
- **공정성 규칙 2 — 독립 정답:** 전이 버킷 (b)의 정답은 그래프를 만든 소스가 아니라 **독립 근거**에서 도출한다 —
  Flockhart 표/공개 DDI 증례를 **비공개로 참조해 손으로 직접 작성한 Q&A 사실**(약물상호작용 사실 자체는 저작권 대상 아님)만 repo에 넣고,
  **Flockhart 표나 그 파생 export는 절대 커밋하지 않는다**(라이선스 하드룰과 충돌 없음). "teaching to the test" 회피. 단일 저자 한계는 README에 명시.
- **지표:** 상호작용 집합 기준 precision/recall/F1, **전이 케이스 적중률(헤드라인)**, query당 latency·token cost.
  RAGAS는 free-text RAG용이므로 그래프 출력엔 **경로를 (subject, relation, object) 문장열로 직렬화**해 faithfulness/citation에 투입하거나,
  **path-level precision/recall**로 대체한다(어느 쪽이든 측정 방법을 README에 적는다).
- **결과는 가설이지 미리 정한 숫자가 아니다.** "벡터가 전이에서 약하다"는 **검증할 가설**로 두고, 공정한 A/B가 내놓는 수치를 그대로 보고한다.
- **정직한 "그래프가 지는 구간" 섹션:** 단일약 조회, 최신·실시간 약, latency 민감 상황 → 벡터/하이브리드가 낫다.
  (이 trade-off 분석이 시니어 판단의 신호. 2025~26 논문들이 GraphRAG 이득 과장을 깠다는 점을 명시.)

## Open Questions / Risks

- **DUR 데이터셋 ID** — 성분레벨 쌍을 주는 게 `15056780`(성분)인지 `15059486`(품목)인지 Day1에 10행 샘플로 확정해 바인딩.
- **성분명 ↔ RxNorm 매핑 커버리지** — 직접 코드맵이 없으므로 이름 정규화+퍼지매칭. 일부 한국 성분은 RxNorm 미존재 가능 →
  매핑 실패율을 측정·문서화(연구급 서브태스크 + 정직성 신호). **타임박스**로 한정 약군에만 적용.
- **openFDA 라벨 CYP 추출 정밀도** — 1순위 작업이므로 작은 골드 추출셋으로 precision 먼저 측정한 뒤 확장.
- **DrugBank/Flockhart는 비공개 QA 전용** — 공개물에 절대 포함 금지(라이선스 하드룰). 추출 결과 검증에만 로컬 사용.
- **스코프 폭주 주의** — 전체 약전이 아니라 한정된 약군으로 시작. 평가표가 핵심이지 그래프 크기가 핵심이 아님.

## The Assignment (다음에 할 한 가지 구체적 행동)

코드 한 줄 쓰기 전에, **오늘 안에**:
1. `data.go.kr`에서 DUR dev key를 발급받고 **`15056780`(성분)와 `15059486`(품목) 둘 다 병용금기 10행을 떠본다** —
   어느 쪽이 `성분A ↔ 성분B + reason` 깨끗한 엣지를 주는지 두 눈으로 확인하고 그 ID를 채택.
2. openFDA 라벨 한 개(예: `simvastatin`)를 `api.fda.gov/drug/label.json`로 받아 **"Drug Interactions/Clinical Pharmacology"에
   CYP3A4 기질이라는 문장이 실제로 있는지** 확인 — 추출 1순위 전략이 성립하는지 검증.
3. amiodarone/clarithromycin/simvastatin 같은 **전이 골드 질문 5개를 정답 경로까지 손으로 적어본다.**
   5개를 못 적으면 평가가 안 서는 거고 = 스코프를 조정해야 한다는 신호.

→ 다음 세션에 **(어느 DUR ID가 맞는지) + openFDA 라벨 샘플 + 손으로 적은 골드질문 5개**를 가져오면, 바로 구현 계획으로 들어간다.

## What I noticed about how you think

- 처음부터 "1. 데이터 수집 가능 2. 실제 수요"라는 **두 기준을 직접 못박고** 시작했다. 대부분은 아이디어부터 말하고
  기준은 안 정한다 — 이 순서를 뒤집은 게 이 세션을 빠르게 수렴시킨 이유다.
- 도메인을 잘못 눌렀을 때 그냥 두지 않고 "**다시 질문해줘 잘못눌렀어**"라고 즉시 바로잡았다. 입력 정확도에 민감하다.
- "**계속 진행**" 그리고 idle에 대한 답답함 — **출하 편향(bias to ship)**이 강하다. 좋은 신호다. 단, 이 프로젝트에서
  그 에너지를 그래프 크기 키우는 데 쓰지 말고 **평가표와 라이브 데모**에 쓰면 포트폴리오가 산다.
